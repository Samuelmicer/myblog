    TileFlow: A Framework for Modeling Fusion Dataflow via Tree-based Analysis¶

    • 作者：郑思泽，梁云
    • 单位：北京大学
    • MICRO'23

    Timeloop是一个数据流性能模型，通过写加速器架构配置，软件计算问题描述，以及软硬件的映射文件（mapping），Timeloop就可以给出一个估计的执行性能。这种性能模型，对于AI加速器设计，数据流映射设计，都是有积极意义的。相关的工作还有MAESTRO (MICRO'20)，TENET (ISCA'21)等等。不过这些数据流性能模型都只关注一个算子的映射问题。一个算子，其实就是一个完美循环，是相对容易数学建模和分析的问题。但是现在我们面对的数据流很少有单个算子的了，大部分情况下都要考虑多个算子融合起来。从现在的AI芯片设计也可以看到，NV GPU有Cuda Core和Tensor Core，一边做MMA计算，一边做Vector运算；NPU也是Matrix Core和Vector Core一起使用。这样，我们做一个MMA+Epilogue的融合就非常自然。近年来由于Transformer架构成为DNN主流，很多融合技术都为self-attention结构和MLP结构做了特殊设计，比如FlashAttention，我之前的HPCA工作Chimera也是这个范畴。所以我们认为，现在的数据流性能分析框架也应该演进了，不能再只考虑一个算子。演进的积极意义，一方面是可以针对将来出现的新的架构，设计数据流的时候，可以提供一些性能模型指导，反过来，对于某一融合数据流，设计专门的硬件架构的时候，也可以用这样的模型去估计一下资源怎么分配能满足性能和功耗需求。TileFlow就是基于这样的想法做出来的框架，我们的分析方法是基于Timeloop的方法的演进和扩展。(作者注)

    3D融合数据流搜索空间¶

    overview¶

    TileFlow的整体工作流程如Figure 3所示，主要由两部分构成：以tile为中心的表示和基于树的分析。作者定义3D的融合数据流搜索空间包括：计算顺序 compute ordering、资源绑定 resource binding、循环分块 loop tiling。

    compute ordering¶

    计算顺序包括节点之间tile的计算顺序(operator orders)和tile内部的循环计算顺序(loop orders)。树的结构适合描述这种计算顺序，节点的外部循环可以形成根节点，内部循环可以形成子节点。硬件的抽象包括了多个内存层级，有L0、L1、L2层级。当算子OP1和OP2进行融合时，在tree中表现为OP1的节点被插入到OP2的子节点中，在L1层面融合，如Figure 4©所示。并且OP1和OP2融合后计算得到的tile，与OP3在L2层级融合。计算顺序即OP1计算完一个tile存到L1 cache里，然后计算OP2，重复这些操作直到OP2在L2的tile计算完成。然后用L2中的tile计算OP3。重复上面的操作，直到OP3计算完。

    resource binding¶

    资源绑定有多种形式。主要讨论层间的资源绑定原语。结合Figure 4(d)和Table 1解释：

    • Sequential: 每个tile计算的时候完全占用所有的计算资源和存储资源。如果下一层用不到，那么计算完的数据要存放到外部。
    • Sharing: 每个tile计算的时候完全占用所有的计算资源，但是存储资源是共享的。
    • Parallel: 不同tile同时计算，使用各自的计算资源和存储资源。
    • Pipeline: tile按照流水线的行为计算，各自有各自的计算资源。

    Pipe形式可以减少延迟，增加吞吐量，代价是更高的计算和内存需求。对于大的tile，Pipe可能没有优势。Para和Pipe类似，但是仅适用于没有数据依赖的情况。Seq对延迟没有改善，但是能节省计算、存储资源，因为一次只能计算一个tile。Share和Seq类似，允许更多数据存在片上，增加数据的locality，但是增加了内存需求。

    Tile-centric Notation¶

    这个简单看看。

    一个内存层级n的tile \(T_n\)可以表示为：

    \[T_n={l_1^n,l_2^n,...}(T_{n-1}^1,T_{n-1}^2,...)\]

    其中\({l_1^n,l_2^n,...}\)是给子tile \((T_{n-1}^1,T_{n-1}^2,...)\)嵌套(增加)的外层循环。下面展示了Figure 4(a)的一个可能的融合数据流的例子：

    \(level 0: T_0^0={i_0,l_o,k}(A), T_0^1={i_0,l_0}(B), T_0^2={i_0,j_0,l_0}(c)\)

    \(level 1: T_1^0={i_1,l_1}(T_0^0, T_0^1), T_1^1={i_1,j_1,l_1}(T_0^2)\)

    \(level 2: T_2^0={i_2,j_2,l_2}(T_1^0, T_1^1)\)

    基于树的分析¶

    数据搬运开销¶

    一类是单个tile内部的数据搬运，根据空间、时间的展开维度，考虑数据重叠与否，计算数据搬运开销。计算的时候考虑的是不同子图之间的差异。

    另一类是tile之间的数据搬运，同样计算子图之间的差异，然后把这些差异累加起来。资源绑定的不同对搬运的计算也有区别。另外还需要考虑不同内存层级的搬运。

    资源占用¶

    遍历树，分析计算、存储占用。公式不打了。

    性能分析¶

    延迟计算是考虑理想情况，如果计算比访存延迟高，就假设访存被完全overlap，延迟等于计算延迟，否则计算被访存overlap，延迟等于访存延迟。

    功耗就是把访存量和计算量传递给已有的能耗评估框架。

    设计空间探索¶

    采用遗传算法和蒙特卡洛树 Monte Carlo Tree Search (MCTS) 搜索结合的方法。

    先对排序树和资源绑定原语进行编码，作为遗传算法的个体。遗传算法可以进行交叉、变异操作，生成分析树的种群。然后把分析树传递给蒙特卡洛树搜索，在约束范围内生成优化的tiling系数，每个蒙特卡洛搜索的结果都对应Figure 7©中展示的编码的tiling表。经过评估之后，结果反馈给MCTS。重复上百次这样的搜索操作，优化的tiling系数被返回给遗传算法，遗传算法获得最佳的tiling系数对应的分析树的最佳性能，保留前K个分析树再生成新的种群。重复这样的迭代数百次，得到最佳的融合数据流。

    实验¶

    作者用Chisel实现了一个四核的加速器，每个核心有两个PE阵列，一个用于矩阵乘(16×16)，一个用于向量计算(16×3)。每个核心有384KB的SRAM。DRAM带宽是256GB/s。Word width(数据位宽？)为16bit。在TSMC 22nm工艺下面积为 \(7.84mm^2\)，频率400MHz。实验比较基本上都是在自己的仿真器上模拟进行的，不一一列举了。
